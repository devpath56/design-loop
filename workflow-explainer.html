A quality harness for design

CI for design taste.

gate: PASS·0 critical·2 checkers agreed

An AI writes the UI. A pipeline of gates decides whether it ships.

It is the discipline you already trust for code, pointed at how a screen looks and reads: a linter, a test suite, a reviewer who did not write the change, and coverage that proves the checks actually ran. Nothing here rests on someone saying it looks fine.

> Written for engineers. No design background needed.

The loop, in five moves

One change in, a graded change out.

You trigger a run on one prototype. The change flows through five stations, left to right. A run that fails is kept, not hidden: the failing row is the most useful one in the table.

01 · MAKE

Generate

A model makes one focused change, steered off the generic defaults that AI design collapses toward.

02 · CHECK

Two checkers

A deterministic pass (contrast, focus, overflow) and a second model that reviews the file cold.

03 · GATE

Binary

Ships only if the deterministic pass is clean and the reviewer logs nothing critical. Three tries, then stop.

04 · LEARN

Record

Every run writes one reusable lesson. A spaced quiz resurfaces it later so it sticks.

05 · PUBLISH

Show

Six views regenerate: the decision table, the side-by-side history, loop health, and a coverage check.

Why it exists

Every part was built to kill a specific recurring failure.

Not features chosen up front. Each mechanism traces to a friction that kept recurring, logged as a numbered failure the way you would file a flaky-test bug. Open one to see the receipt.

01All AI design looks the same. the maker Fix: an off-distribution generator plus 57 named "slop" gates. Slop is the house look a model falls into by default, the same gradient hero and centered everything. Each tell is a named rule the reviewer checks for, so "this looks machine-made" becomes a lint failure with a line number.
plain terms
Like a linter, except the rules are for visual clichés instead of unused variables.
the tell
Structural devices (numbering, dividers, eyebrows) must encode something true, not decorate. A divider that separates nothing is flagged.
receipt
$ hallmark audit page.html
[ { "gate":"decorative-structure", "severity":"major",
  "where":"hero:41", "fix":"the accent bar encodes nothing, cut it" } ]
02The author is the worst reviewer of their own work. cold review Fix: the reviewer is a second model that never saw the reasoning. Run inline, a review grades the argument it just made and waves it through. A fresh reviewer sees only the file, so it catches what the author is blind to.
plain terms
Code review by someone who did not write the change. Two independent judges fail in different ways; that is the point.
the guard
The reviewer runs in a separate process and is handed the file and nothing else. No diff, no rationale, no history.
proof it works
On a later audit, the cold reviewer found a required field that accepted whitespace, and a bug the author had read past three times.
03The build was green, but half the checks never ran. coverage Fix: a coverage step that asks "did each checker actually execute", not just "did it pass". Six runs in a row showed a green gate while the craft checks had never run once. A missing result looked exactly like a clean one.
plain terms
CI passing because the test suite was silently skipped. Absence of a finding is not a finding of absence.
filed as
Bug CF-065, "absence takes the shape of the nearest acceptable outcome." Six distinct instances in one session.
the rule
A check must distinguish three outcomes, never two: it held / it failed / it never ran. A binary check folds the third into the first.
04A test that can never fail is not a test. mutation Fix: every check ships a control that must fire, and the checker itself gets broken on purpose to confirm the test screams. A control wired to real logic and one hardcoded to pass look identical on the page. They diverge only when the code breaks, the one moment the check exists for.
plain terms
Mutation testing. You damage the code and demand the suite go red. If it stays green, the test was theater.
filed as
Bug CF-067. Negative controls were assumed live because they existed. Reading cannot tell a wired control from a fake one.
receipt
# delete a checker's body, re-run
ok  killed by 5 controls
SURVIVED nothing noticed → a hole in the suite
05Rules written in a doc get ignored. enforcement Fix: a rule is real only once a script enforces it. A style guide nobody runs decays to zero. So the house style is executable: contrast thresholds, motion timing, and hierarchy ratios are code that fails the build, not prose in a wiki.
plain terms
A lint rule, versus a paragraph in CONTRIBUTING.md that everyone stopped reading in week two.
the test
Revert any change and ask whether a gate goes red. If it stays green, the change was a suggestion, not a guarantee. There is a one-command tool that proves this.
the boundary
The gate checks the artifact, never the intent. It can prove the field is filled; it cannot prove the answer is honest. That limit is stated, not hidden.
06"Looks off" is not something you can act on. measurement Fix: turn the vague complaint into a number with a threshold. Visual hierarchy gets measured on four levers as ratios, and tap targets are checked against three published standards, each cited with its limits. A code smell becomes a metric.
plain terms
Like a cyclomatic-complexity gate: it replaces "this feels heavy" with "> 1.1x or it fails".
hierarchy
size, weight, contrast, spacing. Measured as the difference between levels, never the raw values. 12/13/14px is three numbers and one rank.
receipt
size steps  12 / 14 / 20px → 1.17x · 1.43x ok
levers     4/4 in play (size · weight · contrast · spacing)

Anatomy

Six parts, each replaceable.

The loop is built from named pieces, the same way a CI system is a trigger plus stages plus a config file. Each is a script you can read, swap, or delete, and the coverage step will tell you if you deleted one that mattered.

piecewhat it isthe code analogue
triggerYou, running one command on one prototype.git push
skill fileThe house style: contrast floors, motion timing, hierarchy ratios..eslintrc
makerThe model that makes one off-distribution change.the codegen step
checkersDeterministic a11y pass + a cold structural reviewer.unit tests + code review
gateBinary verdict with teeth. Clean + no critical, or it stops.required status check
stateA ledger of runs and lessons, resurfaced on a spaced schedule.CHANGELOG + postmortems

The receipts are the product

Nine graded runs, twelve reusable lessons, a catalog of 66 named failure modes. The value is not any single screen; it is the trail that makes the next screen faster and the checkers themselves harder to fool.

Two judges, decorrelated

The deterministic checker catches what code can measure exactly (a contrast ratio is not a matter of opinion). The cold model catches what needs judgment. Neither grades its own work, so their blind spots do not overlap.

Where it stops, honestly

What a gate cannot do.

  • It checks the artifact, not the intent. A gate can prove a field is filled and a contrast ratio clears. It cannot prove the design solves the right problem. That call stays human.
  • Coverage is relative to the checks that exist. A file no checker looks at reads as clean because nothing measured it. Green means the checks that ran, passed, and nothing more.
  • Automated taste can be confidently wrong. A fast, valid signal is what turns practice into skill; a fast, invalid one just trains you to trust bad calls faster. So every judgment is sourced or labeled opinion.
